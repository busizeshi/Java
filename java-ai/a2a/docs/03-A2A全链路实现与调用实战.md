# 文档三：手搓 A2A Server + Client + Orchestrator 全链路实战（含代码映射）

## 1. 全链路目标

目标是构建一个最小可运行的 A2A 协作闭环：

1. 专业 Agent 对外暴露 Skill
2. 编排 Agent 自动发现能力并做路由
3. 通过 A2A 协议发起任务并获取结果

当前工程以“销售分析”为领域，包含两个 Skill：

1. 销售汇总查询
2. 销售异常分析

## 2. 服务端实现：A2A Server 的最小能力集合

最小集合包含两个入口：

1. `GET /.well-known/agent.json`：能力发现
2. `POST /`：JSON-RPC 任务入口

`Agent Card` 已在前两篇文档展开，本篇聚焦任务执行链路。

## 3. JSON-RPC 统一入口与方法分发

来源文件：[A2aController.java](D:\jwd-dev\study\Java\java-ai\a2a\src\main\java\com\jwd\controller\A2aController.java)

```java
@PostMapping(value = "/", consumes = "application/json", produces = "application/json")
public JsonRpcResponse handle(@RequestBody JsonRpcRequest request) {
    return switch (request.method()) {
        case "tasks/send"   -> handleTaskSend(request);
        case "tasks/get"    -> handleTaskGet(request);
        case "tasks/cancel" -> handleTaskCancel(request);
        default -> JsonRpcResponse.error(-32601, "Method not found: " + request.method(), request.id());
    };
}
```

说明：

1. 所有 A2A 方法通过同一 HTTP 入口承载
2. 通过 `method` 做协议层分发，便于后续扩展 `tasks/sendSubscribe`
3. 未知方法返回 JSON-RPC 标准错误码 `-32601`

## 4. tasks/send：任务创建与执行主流程

### 4.1 执行步骤

`handleTaskSend` 的执行顺序如下：

1. 解析 `id/sessionId/skillId/history`
2. 将任务先落为 `SUBMITTED`
3. 按 `skillId` 定位 `SkillHandler`
4. 执行 skill 获得完成态 `Task`
5. 保存并返回结果

这条流程体现了 A2A 的核心思想：任务先入库、再执行、再回写状态。

### 4.2 状态前置保存的意义

先保存 `SUBMITTED` 有两个直接收益：

1. 即使后续执行失败，也保留了任务存在性
2. 可扩展为异步模式（后续转 `WORKING/COMPLETED/FAILED`）

### 4.3 history 解析

`parseHistory` 将 JSON 结构转为 `List<Message>`，并支持 `text/data` Part。

说明：

1. 统一的消息结构使 Skill 实现不关心底层传输格式
2. 为后续引入 `file` Part 预留了扩展点

## 5. tasks/get 与 tasks/cancel：任务管理能力

### 5.1 tasks/get

按任务 ID 查询当前状态。

价值：

1. 支持调用方轮询长任务
2. 支持服务重试与故障恢复

### 5.2 tasks/cancel

将任务更新为 `CANCELED`。

价值：

1. 在长任务场景中支持撤销
2. 为资源控制和用户交互保留控制面

## 6. 客户端实现：A2A Client 封装调用细节

来源文件：[A2aClient.java](D:\jwd-dev\study\Java\java-ai\a2a\src\main\java\com\jwd\client\A2aClient.java)

当前封装了 4 个核心操作：

1. `getAgentCard(agentUrl)`
2. `sendTask(agentUrl, taskId, sessionId, skillId, history)`
3. `getTask(agentUrl, taskId)`
4. `cancelTask(agentUrl, taskId)`

说明：

1. 调用层不再手写 JSON-RPC payload
2. 错误统一在 `postJsonRpc` 中检查并抛出异常
3. 便于后续扩展认证头、重试、超时控制

## 7. 编排层实现：Orchestrator 决策与委托

来源文件：[OrchestratorAgent.java](D:\jwd-dev\study\Java\java-ai\a2a\src\main\java\com\jwd\client\OrchestratorAgent.java)

核心职责：

1. 收集可用 Agent 能力摘要
2. 使用 LLM 生成路由决策 JSON
3. 根据 `agentName + skillId` 发起 A2A 调用
4. 提取任务结果文本返回上层

### 7.1 路由决策输入

来自 `AgentRegistry.buildAgentsSummary()` 的能力摘要通常包含：

1. Agent 名称
2. URL
3. Agent description
4. Skill 列表与 description

能力摘要质量越高，路由越稳定。

### 7.2 路由决策输出

通过 system prompt 约束输出为固定 JSON：

1. `agentName`
2. `skillId`
3. `reason`

这是“自然语言理解”到“可执行动作”的桥接层。

### 7.3 当前实现的鲁棒性风险

`parseDecision` 使用字符串截取解析 JSON，存在脆弱性：

1. 模型输出格式轻微变化可能导致解析失败
2. 转义字符或字段顺序变化可能误判

建议升级为 `ObjectMapper.readValue` 严格反序列化。

## 8. 对外接口：编排入口控制器

来源文件：[OrchestratorController.java](D:\jwd-dev\study\Java\java-ai\a2a\src\main\java\com\jwd\controller\OrchestratorController.java)

说明：

1. 对外暴露统一编排入口

## 10. 端到端调用时序（实战）

以请求“帮我分析一下本周销售有没有异常”为例：

1. 调用 `GET /api/orchestrator/ask?request=...`
2. 编排器读取 Agent 能力摘要
3. LLM 产出路由：`analyze-sales-anomaly`
4. `A2aClient.sendTask` 发送 `tasks/send` 到 `POST /`
5. `A2aController` 分发到 `SalesAnomalySkillHandler`
6. Skill 生成分析并返回 `Task(COMPLETED + Artifact)`
7. 编排器提取文本并返回给调用方

## 11. 调试与故障定位

## 11.1 先验检查顺序

1. `/.well-known/agent.json` 是否可访问
2. Skill id 是否与处理器 `skillId()` 完全一致
3. `tasks/send` payload 是否包含 `history.parts`
4. 路由 JSON 是否满足预期格式

## 11.2 常见错误

1. `Unknown skill`：Card 中 skill id 与 Handler 不一致
2. `Task not found`：查询了未创建或已清理的 task id
3. 路由失败：LLM 输出未满足 JSON 约束

## 11.3 观测建议

1. 为每次 `tasks/send` 打印 `taskId/skillId/sessionId`
2. 记录状态迁移日志（SUBMITTED -> COMPLETED/FAILED）
3. 对路由原始输出做审计日志（注意脱敏）

## 12. 向企业级演进的升级路线

## 12.1 协议能力升级

1. 增加 `tasks/sendSubscribe`（SSE）
2. 增加 `pushNotifications` 回调
3. 支持 `INPUT_REQUIRED` 交互闭环

## 12.2 稳定性升级

1. Task 存储迁移到 Redis/DB
2. Client 增加重试与超时策略
3. 路由 JSON 使用严格反序列化

## 12.3 工程治理升级

1. Agent/Skill 版本化管理
2. 统一错误码与可观测指标
3. 提供集成测试覆盖典型编排链路

## 13. 本文档知识要点小结

1. A2A Server 的核心是“统一入口 + 任务状态机 + Skill 分发”
2. A2A Client 的核心是“发现、调用、查询、取消”的协议封装
3. Orchestrator 的核心是“能力摘要 + LLM 路由 + 委托执行”
4. 最小同步闭环已经具备，下一步重点是异步流式与鲁棒性增强
