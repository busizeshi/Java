# 文档一：A2A 基础原理与核心协议模型（含代码映射）

## 1. A2A 要解决的问题

在企业多 Agent 场景中，常见结构是“一个编排 Agent + 多个领域 Agent”。

示例任务：安排客户拜访（机票、酒店、邮件、CRM 更新）。
这个任务涉及多个系统、多个能力域，通常由不同团队、不同框架、不同语言实现。

核心痛点不是“工具怎么调用”，而是“Agent 怎么协作”：

1. 如何发现其他 Agent 的能力边界
2. 如何统一调用格式
3. 如何追踪任务状态
4. 如何处理长任务、失败、取消、补充信息

A2A（Agent2Agent Protocol）用于标准化这些问题。

## 2. A2A 与 MCP 的分工关系

A2A 与 MCP 不是替代关系，而是分层协作关系。

1. MCP：`Agent ↔ Tool/Data`（工具调用层）
2. A2A：`Agent ↔ Agent`（协作编排层）

工程上可理解为：

1. 上层编排通过 A2A 委托任务给专业 Agent
2. 专业 Agent 内部再通过 MCP 访问数据库/API/文件等工具

## 3. A2A 协议栈与通信方式

A2A 常见协议组合：

1. 语义层：Task、Skill、Message、Artifact
2. 编码层：JSON-RPC 2.0
3. 传输层：HTTP/HTTPS
4. 推送层：SSE（`tasks/sendSubscribe`）

最小可用实现通常包含：

1. `GET /.well-known/agent.json`：能力发现
2. `POST /`：JSON-RPC 统一入口（如 `tasks/send`、`tasks/get`）

## 4. Agent Card：能力发现入口

Agent Card 固定发布路径：`/.well-known/agent.json`。
调用方通过这个 JSON 判断：是否调用该 Agent、调用哪个 Skill、采用哪种通信模式。

### 4.1 代码映射：Agent Card 接口

```java
@GetMapping("/.well-known/agent.json")
public Map<String, Object> agentCard() {
    return Map.of(
        "name", "销售数据分析 Agent",
        "description", "专门处理销售数据分析任务，能生成报表、识别异常、预测趋势",
        "url", "http://localhost:8080",
        "version", "1.0.0",
        "capabilities", Map.of(
            "streaming", false,
            "pushNotifications", false
        ),
        "defaultInputModes", List.of("text"),
        "defaultOutputModes", List.of("text", "data"),
        "skills", buildSkills()
    );
}
```

说明：

1. `description` 是“路由语义”，决定编排 Agent 是否把任务分配过来
2. `capabilities` 决定客户端调用策略（同步/流式/回调）
3. `skills` 是能力清单，供 LLM 或编排器进行技能匹配

### 4.2 Skill 声明样例

同文件中的 `buildSkills()` 返回两个能力：

1. `query-sales-summary`
2. `analyze-sales-anomaly`

每个 Skill 都包含 `id/name/description/tags/examples/inputModes/outputModes`。

说明：

1. `id` 是执行路由主键
2. `description` 定义适用场景，避免“能力边界模糊”
3. `examples` 用于消歧，帮助 LLM 识别相似技能差异

## 5. Task 模型：A2A 的核心执行单元

A2A 与普通函数调用的关键差异：Task 是有状态对象。

### 5.1 代码映射：Task record

```java
public record Task(
    String id,
    String sessionId,
    TaskStatus status,
    List<Message> history,
    List<Artifact> artifacts,
    Map<String, Object> metadata
) {}
```

说明：

1. `id/sessionId`：任务与会话标识
2. `status`：状态机快照（如 SUBMITTED/COMPLETED/CANCELED）
3. `history`：输入上下文
4. `artifacts`：可交付产物
5. `metadata`：扩展字段

### 5.2 代码映射：TaskState

```java
public enum TaskState {
    SUBMITTED,
    WORKING,
    INPUT_REQUIRED,
    COMPLETED,
    CANCELED,
    FAILED
}
```

说明：

1. 这组状态可以表达同步和异步生命周期
2. `INPUT_REQUIRED` 是 A2A 典型特性：任务中途向上游补充要件

## 6. Message 与 Part：多模态消息容器

A2A 的输入输出不是只有纯文本，采用“消息 + 分片”结构。

### 6.1 代码映射：Part 结构

```java
@JsonInclude(JsonInclude.Include.NON_NULL)
public record Part(
    String type,   // "text", "data", "file"
    String text,
    Map<String, Object> data
) {
    public static Part text(String text) {
        return new Part("text", text, null);
    }
    public static Part data(Map<String, Object> data) {
        return new Part("data", null, data);
    }
}
```

说明：

1. `text`：自然语言内容
2. `data`：结构化 JSON（便于程序处理）
3. `file`：协议概念已预留，当前实现尚未落地文件字段

## 7. Artifact：任务交付物

Artifact 表示任务完成后交付给调用方的成果，可包含多个 Part。

工程实践建议：

1. 同时输出 `text + data`，兼顾可读性与可计算性
2. 在 `data` 中保留关键业务字段（如 severity、timeRange、confidence）

## 8. JSON-RPC 承载 A2A 语义

A2A 服务端通过 `POST /` 接收 JSON-RPC 请求，根据 `method` 分发。

### 8.1 代码映射：统一入口

```java
@PostMapping(value = "/", consumes = "application/json", produces = "application/json")
public JsonRpcResponse handle(@RequestBody JsonRpcRequest request) {
    return switch (request.method()) {
        case "tasks/send"   -> handleTaskSend(request);
        case "tasks/get"    -> handleTaskGet(request);
        case "tasks/cancel" -> handleTaskCancel(request);
        default -> JsonRpcResponse.error(-32601, "Method not found: " + request.method(), request.id());
    };
}
```

说明：

1. `tasks/send`：发起任务
2. `tasks/get`：查询状态
3. `tasks/cancel`：取消任务

## 9. 本文档知识要点小结

1. A2A 关注 Agent 间协作标准化，MCP 关注 Agent 与工具标准化
2. Agent Card 是发现入口，Skill 是路由依据
3. Task 是有状态执行对象，是 A2A 区别于普通工具调用的核心
4. Message-Part 与 Artifact 保证“可读 + 可计算”的交付结构
5. JSON-RPC + HTTP 使协议落地简单且语言无关
